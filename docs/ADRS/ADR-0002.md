# ADR-0002: Effect-TS for data fetching and validation

## Status

Accepted

## Context

The MSW mock returns 2000 products as JSON. We need to fetch, validate the shape, and surface errors to the UI. Options considered:

- **Plain fetch + manual validation** — simple, no deps
- **Zod** — popular schema validation
- **Effect-TS** — typed effects with structured errors, schema validation, fiber-based cancellation

## Decision

Use Effect-TS (`effect` v3 package) with `Schema.Class` for product validation, `Effect.gen` for the fetch pipeline, and `Context.Tag` + `Layer` for dependency injection of `ProductService`.

## Consequences

- **Positive:** `Schema.decodeUnknown` validates API responses at runtime with zero boilerplate. `FetchError` and `DecodeError` as `Schema.TaggedError` give typed, discriminated error handling.
- **Positive:** `Effect.runFork` + `Fiber.interrupt` in the React hook provides proper cancellation on unmount — the HTTP request is actually interrupted, not just ignored.
- **Positive:** `Effect.match({ onSuccess, onFailure })` handles errors type-safely inside the pipeline, keeping Effect internals out of the React layer.
- **Negative:** Effect is a large dependency (~150KB). Justified by the structured error handling and cancellation semantics that would otherwise require significant manual code.
- **Negative:** Steeper learning curve for developers unfamiliar with Effect patterns.
