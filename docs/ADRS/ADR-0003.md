# ADR-0003: Lazy MSW initialization for Module Federation

## Status

Accepted

## Context

The scaffold auto-starts MSW in `remote-products/src/msw/browser.ts` via a side-effect import in `main.tsx`. This works when the remote runs standalone, but breaks when the host loads `ProductList` via Module Federation — the host's `main.tsx` never imports the MSW browser module, so the service worker is never registered.

## Decision

Extract MSW initialization into `msw/initMsw.ts` with an `ensureMsw()` function that lazily starts the worker once. Call `ensureMsw()` from the `useProducts` hook before making any fetch calls. Remove the auto-start side effect from `browser.ts`.

```ts
let started = false;
export async function ensureMsw(): Promise<void> {
  if (started) return;
  started = true;
  const { worker } = await import("./browser");
  await worker.start({ quiet: true, ... });
}
```

## Consequences

- **Positive:** MSW works regardless of entry point — both standalone remote dev and host-via-MF contexts initialize the worker before the first API call.
- **Positive:** The `started` guard ensures the worker is only started once even if multiple components call `ensureMsw()`.
- **Positive:** `mockServiceWorker.js` must be copied to the host's `public/` directory since the service worker registers relative to the serving origin.
- **Negative:** Adds ~2s to first render (MSW start + mock API delay). Acceptable for development mocks.
